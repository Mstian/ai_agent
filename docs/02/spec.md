# MewCode 第二阶段 · 工具系统 Spec

## 背景

第一阶段实现了纯对话能力：用户输入问题 → 调用大模型 API → 流式输出回复。但模型只能"动嘴"，无法读文件、搜索代码、执行命令。

第二阶段给 agent 装上工具系统。用户提问后，模型不再只能动嘴，而是能识别自己需要用哪个工具，代码去执行，再把结果喂回去，模型据此生成回答。从聊天机器人变成真正能干活的 agent。

## 目标

- 建立统一的 Tool 接口体系，每个工具实现它
- 首批实现六个核心工具：读文件、写文件、改文件、执行命令、按模式找文件、搜代码内容
- 一个注册中心集中管理工具，按名查找，转成 API 识别的工具列表
- Provider 层能解析流式工具调用（JSON 参数碎片拼接），执行完把结果回灌进对话历史
- 不做工具连环调用，模型拿到一次工具结果就生成回复停下

## 功能需求

### F1: 统一工具接口
- 每个工具声明 name、description、parameters（JSON Schema 格式）、execute 方法
- execute 接收参数和上下文信息，返回成功/失败和输出内容
- 工具执行带超时和错误处理，失败信息包成结构化结果返回给模型，让模型能调整而不是崩掉

### F2: 六个核心工具
- **读文件 (read_file)**：按路径读取文本，防路径逃逸，限 1MB，检测二进制拒绝
- **写文件 (write_file)**：创建/覆盖文件，递归建父目录，防路径逃逸
- **改文件 (edit_file)**：原文唯一匹配替换。匹配 0 次 → 报未找到；≥2 次 → 报匹配 N 处；1 次 → 执行替换
- **执行命令 (run_command)**：执行 shell 命令，硬编码危险模式黑名单（rm -rf /、fork bomb 等），超时机制，输出截断
- **搜索文件 (glob)**：按模式查找文件，支持 *、**、?，忽略 node_modules 等
- **搜索代码 (grep)**：正则/文本搜索文件内容，返回文件名、行号、匹配行

### F3: 工具注册中心
- ToolRegistry 集中登记工具，按名查找
- toAPIFormat() 把已注册工具转成 API 识别的工具列表格式
- 重名注册报错

### F4: 流式工具调用解析
- Anthropic：解析 content_block_delta 中的 input_json_delta 碎片，在 content_block_stop 时拼装完整 JSON 参数
- OpenAI：解析 choices[0].delta.tool_calls 增量，按 index 归并 id/name/arguments
- 两种 API 的差异由 Provider 层屏蔽，上层只看到统一的 ToolUseEvent

### F5: 工具执行与结果回灌
- ChatManager 检测 assistant 回复中包含 tool_use block 时，逐个执行工具
- 工具不存在时构造错误 tool_result
- 正常执行的 tool_result 注入对话历史
- 工具执行完后，再调一次模型（不带 tools）生成最终文字回复
- 不做连环调用（如果第二次又返回 tool_use，不再执行）

### F6: 工具调用视觉效果
- 模型发起工具调用时，终端显示灰色提示：⚡ 准备调用
- 执行中显示：🔧 正在执行 + 参数摘要
- 完成显示：✅ 完成 / ❌ 失败

## 非功能需求

### N1: 可扩展性
- 新增一个工具只需实现 Tool 接口 + 在 registry 注册
- 新增一个 Provider 的 tool_use 解析只需在对应 Provider 中实现
- Provider 接口的 streamChat 方法 tools 参数为可选，不传则模型不使用工具

### N2: 安全性
- 文件读写工具防路径逃逸（../ 不允许超出项目目录）
- 命令执行工具硬编码黑名单拦截危险命令
- 所有工具超时保护（默认 30s）

### N3: 可靠性
- 工具执行异常不导致程序崩溃，转为结构化错误返回给模型
- 模型收到错误后可自行调整参数重试
- 大输出自动截断，避免 token 浪费

### N4: 可测试性
- 每个工具可独立测试（传入临时目录，不依赖真实项目文件）
- Provider 层用 mock fetch 测试 tool_use 流式解析

## 不做的事

- ❌ 工具连环调用（模型拿到结果后再次调用工具再拿结果再调用...）
- ❌ 用户授权确认（危险命令直接拒绝，不弹确认框）
- ❌ 工具并行执行（多个 tool_use 串行执行）
- ❌ 工具结果持久化或缓存
- ❌ 运行时动态注册/卸载工具
- ❌ 非 Anthropic/OpenAI 协议的 tool use

## 验收标准

- AC1: 用户输入"帮我看看 src/main.ts 的内容"，模型调用 read_file 工具，终端显示工具调用视觉效果，最终展示文件内容
- AC2: 用户输入"在 src 目录搜索包含 main 的文件"，模型调用 grep 工具，返回匹配结果
- AC3: 用户不带工具意图的普通对话仍然正常工作
- AC4: 模型请求不存在的工具时，返回"未知工具"错误而不崩溃
- AC5: 模型调用 edit_file 且 old_string 匹配多处时，返回错误说明匹配了几处
- AC6: 危险命令（如 rm -rf /）被安全检查拒绝
- AC7: 切换到 openai 协议，工具调用同样正常工作
- AC8: 所有 66 个测试（原有 + 新增）通过
- AC9: npx tsc --noEmit 零类型错误
