# MewCode 第四阶段 · 结构化系统提示 Spec

## 背景

目前 MewCode 的全局指令只有三行：

```
你是 MewCode，一个终端 AI 编程助手。用中文回答用户的问题。回答简洁、准确。
你可以使用工具来读取文件、搜索代码、执行命令、编辑文件。
对于复杂任务，先规划再分步执行。
```

这导致几个问题：
- 模型经常不遵守关键约定（如"编辑前先读文件"、"优先用专用工具而非 shell 命令"）
- 所有内容混在一起，无法区分稳定指令和变化内容，缓存利用率低
- 新增指令只能往这个字符串里硬塞，没有模块化结构
- 模式切换（plan/full）的信息没有传递给模型，模型不知道当前处于什么模式

第四阶段把系统提示工程化：从"一段话"升级为"结构化的多层提示系统"。

## 目标

- 按职责拆成模块，按优先级拼装，方便插拔
- 区分稳定内容（走缓存）和变化内容（走消息通道），降低 token 成本
- 通过双重强化提高模型对关键规则的遵守率
- 支持运行时注入指令而不污染缓存
- 通过 Anthropic 的缓存命中字段验证策略是否生效

## 功能需求

### F1: 模块化系统提示

把系统提示按职责拆成独立模块，每个模块是一段职责明确、可独立修改的文本。模块按优先级从前往后排：

**固定模块（7 个，按优先级）：**
1. **身份（Identity）** — 名字、角色、能力范围
2. **系统约束（Constraints）** — 工作目录、权限边界、不能做的事
3. **任务模式（TaskMode）** — 当前处于什么模式（规划/执行），该模式下的行为规则
4. **动作执行（Actions）** — 如何执行任务：先规划再动手、分步执行、验证结果
5. **工具使用（ToolUse）** — 工具选择优先级、关键规则（如"编辑前必先读"、"优先用专用工具"）
6. **语气风格（Tone）** — 语言、简洁度、格式偏好
7. **文本输出（Output）** — Markdown 支持、代码块格式、错误报告格式

**可选模块（3 个，按需拼接）：**
8. **自定义指令（CustomInstructions）** — 用户配置的额外指令（后续从文件加载）
9. **已激活 Skill（ActiveSkills）** — 当前启用的特殊能力
10. **长期记忆（LongTermMemory）** — 跨会话的用户偏好和项目信息

**拼装规则：**
- 每个模块有 `key`、`priority`、`content`、`enabled` 属性
- 按 priority 从低到高排序，模块间空行分隔
- 最终 system prompt = 所有 enabled 模块的 content 拼接

### F2: 稳定与变化内容分离

区分两类内容，走不同通道：

| 内容类型 | 通道 | 说明 |
|---------|------|------|
| 稳定指令（固定模块 1-7 的静态部分） | 顶层 system 参数 | 可被 Anthropic Prompt Cache 缓存 |
| 工具描述（tools 数组） | API tools 字段 | 也可被缓存 |
| 变化指令（模式切换、运行时注入） | system 角色消息 | 插入 messages 数组，不影响缓存 |
| 环境信息（cwd、git 状态、日期） | system 角色消息 | 随对话变化，不能缓存 |
| 对话历史（user/assistant/tool） | messages | 正常消息流 |

这样每次请求中，大块的稳定指令和工具描述走缓存（省 token 和时间），只有变化的小块内容需要重新处理。

### F3: 关键规则双重强化

对"模型容易忽略但很重要"的规则，同时在两个地方强调：

| 关键规则 | 工具描述中 | 全局指令中 |
|---------|-----------|-----------|
| 编辑前必先读文件 | edit_file 和 write_file 的 description 第一句 | 工具使用模块中明确列出 |
| 优先用专用工具而非 shell | 每个工具 description 说明适用场景 | 工具使用模块中强调 |
| 路径必须在项目范围内 | read_file / write_file 的 description | 系统约束模块中声明 |
| 先规划再执行 | — | 动作执行模块 + 任务模式模块 |

双重强化的原理：模型在"选择工具"时主要看工具描述，在"决定要不要用工具"时看全局指令。两处都写能覆盖两种决策场景。

### F4: 运行时指令注入

用 role='system' 的消息在运行时注入补充指令。和顶层 system 参数不同，system 角色消息：
- 插入到 messages 数组的适当位置（通常在最新 user 消息之前）
- 不破坏顶层 system 参数的缓存
- 模型不会回复 system 消息
- 可以按轮次动态调整内容

system 消息内容带特殊标记以区分来源：
```
[SystemInstruction]
type: <instruction_type>
---
<instruction_content>
```

instruction_type 包括：
- `mode_switch` — 模式切换通知
- `environment` — 环境信息（cwd、git 状态等）
- `reminder` — 关键规则提醒
- `custom` — 用户自定义指令

### F5: 会话级开关的轮次注入

对于需要跨轮次保持的会话级开关（如 plan mode），通过 system 消息按频率注入：

- **首轮（turn=1）：** 注入完整指令，详细说明当前模式的规则
- **间隔轮次（每 3 轮）：** 重复完整指令，防止模型遗忘
- **其余轮次：** 注入一行精简提醒（如 `[提醒] 当前仍处于规划模式，只能使用只读工具`）

注入在每轮 LLM 调用前执行，由 Agent 或 PromptManager 负责。

### F6: 缓存命中监控

用 Anthropic API 返回的 usage 字段监控缓存效果：
- `cache_creation_input_tokens` — 本次创建缓存的 token 数
- `cache_read_input_tokens` — 本次命中缓存的 token 数
- `input_tokens` — 总输入 token 数

缓存命中率 = cache_read_input_tokens / (cache_read_input_tokens + 非缓存 input tokens)

在流式响应中解析这些字段，在 turn_end 或 agent_done 事件中暴露。

### F7: 定性评估场景

准备一组典型场景做人工对比（不追求自动化）：
1. 用户说"帮我改一下 src/main.ts 的 main 函数" → 模型应先 read_file，再 edit_file
2. 用户说"找到所有 ts 文件" → 模型应用 glob，而不是 find 命令
3. 用户说"帮我重构项目" → 模型应先给计划，再动手
4. 用户处于 plan mode 时输入"创建文件" → 模型应拒绝并提醒处于规划模式

## 非功能需求

### N1: 可扩展性
- 新增模块只需定义 key、priority、content，自动参与拼装
- 模块 content 支持简单的模板变量（如 `{{cwd}}`、`{{mode}}`），运行时替换

### N2: 缓存友好
- 固定模块 content 尽量稳定（不含变量部分），变量部分通过 system 消息注入
- 工具描述保持稳定，不与模式切换联动变化

### N3: 可观测性
- turn_end / agent_done 事件携带缓存命中数据
- 模块拼装结果可通过 debug 模式查看

## 不做的事

- ❌ 项目指令文件加载（如 .mewcoderules、CLAUDE.md）
- ❌ 自动记忆（自动从对话中提取用户偏好并写入长期记忆）
- ❌ 真实 MCP 接入（仅限目前的手动工具系统）
- ❌ 自动化评估（用 LLM-as-judge 跑基准测试）
- ❌ 多语言系统提示（当前仅中文）
- ❌ 动态优先级调整（模块优先级固定，不会根据场景自动调整）

## 验收标准

- AC1: 模型被要求改文件时，自动先 read_file 再 edit_file（而非直接编辑）
- AC2: 模型被要求搜文件时，优先用 glob 工具而非 find 命令
- AC3: 模型被要求做复杂任务时，先给计划再逐步执行
- AC4: plan mode 下模型严格遵守只读约束，拒绝修改文件
- AC5: 用 Anthropic 协议时，turn_end 事件中包含缓存命中数据
- AC6: 非首轮的 plan mode 提醒以精简形式注入（一行而非完整段落）
- AC7: 所有现有测试继续通过，类型检查零错误
- AC8: 模块可以独立启用/禁用，不影响其他模块
