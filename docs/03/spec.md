# MewCode 第三阶段 · Agent Loop Spec

## 背景

第二阶段建好了工具系统——6 个工具、注册中心、流式 tool_use 解析、单轮执行后生成回复。但 agent 只走一轮：模型调一次工具，拿到结果生成回复，就停了。

如果任务需要"读文件 → 分析内容 → 修改代码 → 运行测试 → 发现错误 → 再修改"，用户必须手动一步步引导，每次输入"继续"才能推进。这不是 agent，是带工具调用的 chatbot。

第三阶段给 MewCode 装上 Agent Loop。Agent 拿到任务后自主循环"想 → 做 → 看结果 → 调整"，直到任务完成。从被动应答变成真正自主执行的 agent。

## 目标

- 实现 ReAct 模式（Reasoning + Acting）的自主循环
- Agent 像人一样：拿到任务 → 思考下一步做什么 → 调用工具 → 观察结果 → 继续思考或给出最终答案
- 提供异步事件流解耦 Agent 内核和界面
- 支持 Plan Mode：先只读规划，用户确认后再全工具执行

## 功能需求

### F1: ReAct 循环引擎

Agent 拿到用户任务后进入循环：

```
用户任务
  │
  ▼
┌─────────────────────────────────┐
│ ① 调 LLM（带工具定义）          │
│ ② 模型返回：说完 / tool_use     │
│ ③ 如果是 tool_use → 执行工具    │
│ ④ 工具结果写回对话历史           │
│ ⑤ 回到 ①                       │
│ 模型返回纯文本 → 循环结束        │
└─────────────────────────────────┘
```

每次循环称为一个 turn。循环持续直到满足停止条件。

### F2: 五种停止条件

循环在以下任一条件满足时停止：

1. **模型停止调工具** — LLM 返回纯文本回复（不含 tool_use），表示任务完成或无法继续
2. **到达迭代上限** — 达到最大迭代次数（默认 5 次），自动停止。将当前状态总结给用户，说明"达到迭代上限，以下是已完成的工作"
3. **用户取消** — 用户按 Ctrl+C 中断（和第一阶段行为一致），保留已完成的工作
4. **连续未知工具** — 模型连续 2 次调用不存在的工具，终止循环并报错
5. **流出错** — Provider 返回 error 事件时终止

### F3: 异步事件流

Agent 内核和界面之间通过一套统一的异步事件流通信，实现完全解耦：

事件类型：
- `text_delta` — 文本增量（流式输出到界面）
- `thinking_delta` — 思考过程增量
- `tool_use` — 模型发起工具调用
- `tool_executing` — 工具开始执行
- `tool_result` — 工具执行完成
- `turn_start` — 新一轮循环开始（携带 turn 序号）
- `turn_end` — 一轮循环结束（携带 Token 用量等摘要）
- `agent_done` — Agent 整体执行完成（携带总 turns 数、总 token 用量）
- `error` — 错误事件

界面层只消费事件流，不感知 Agent 内部循环逻辑。

### F4: 流式收集器（双路）

每轮 LLM 调用产出的事件需要同时走两条路：

- **实时通道**：text_delta / thinking_delta 直接推到界面，用户看到逐字输出
- **累积通道**：所有 delta 拼接成完整 ContentBlock[]，用于：
  - 判断 assistant 回复中是否包含 tool_use
  - 完整的 assistant 消息写入对话历史

相当于一个 `StreamCollector`：一边透传文本给界面，一边在后台攒完整响应。

### F5: 多工具并发与串行

模型一次可能返回多个 tool_use block。按安全性分批执行：

- **只读工具**（read_file、glob、grep）：无副作用，可并发执行（Promise.all）
- **有副作用工具**（write_file、edit_file、run_command）：必须按顺序串行执行，不能并发

如果一次返回中既有只读又有副作用工具，分两阶段：先并发跑完只读，再串行跑有副作用的。

每个 tool_use block 携带唯一 ID，tool_result 通过 tool_use_id 对应回去。结果按原顺序写回对话历史。

### F6: Plan Mode 两段式

用户可以通过命令切换 Agent 的工作模式：

- `/plan` — 进入规划模式。只提供只读工具（read_file、glob、grep），Agent 只能查看代码、搜索文件，不能修改任何东西。模型先调研项目，给出执行计划
- `/do` — 进入执行模式。恢复全部工具，Agent 基于之前讨论的计划开始真正执行

Plan Mode 流程：
1. 用户输入 `/plan` → Agent 切换为只读模式
2. 用户描述任务 → Agent 调研代码库 → 生成计划 → 展示给用户
3. 用户审核计划 → 输入 `/do` 确认执行 → Agent 切换为全工具模式
4. Agent 基于计划执行 → 直到完成或用户干预

`/plan` 和 `/do` 之间用户可能需要多次对话来完善计划（在只读模式下讨论），不限制轮数。

### F7: 内置命令扩展

在原有 `/exit`、`/clear` 基础上新增：

- `/plan` — 切换到规划模式（只读工具）
- `/do` — 切换到执行模式（全部工具）
- `/stop` — 停止当前 Agent 循环（不影响对话历史）

模式状态在提示符中体现（如 `[plan] >` 或 `[do] >`）。

## 非功能需求

### N1: 代码解耦
- Agent 循环逻辑独立于界面，可以脱离 TUI 单独运行
- 事件流是 Agent 和界面的唯一通信方式
- 界面可替换（当前是 readline，未来可以是 WebSocket 或 HTTP SSE）

### N2: 可靠性
- 单个工具执行失败不中断整个循环（错误作为 tool_result 返回给模型）
- 迭代上限兜底，防止无限循环耗尽 token
- LLM 调用失败（网络错误、API 错误）终止循环并报告

### N3: 可观测性
- 每轮循环的 token 用量在 turn_end 事件中暴露
- 总 token 用量在 agent_done 中汇总
- 循环序号在 turn_start 中可见

### N4: 性能
- 只读工具并发执行，减少多工具调用时的等待时间
- 流式收集器的累积通道不阻塞实时通道

## 不做的事

- ❌ 权限系统（工具执行前弹确认框让用户 approve）
- ❌ 上下文压缩（对话历史过长时自动裁剪或摘要）
- ❌ 用户交互式确认（"即将执行 rm -rf X，是否继续？[y/N]"）
- ❌ 子 Agent 派生（Agent 内部再创建 Agent）
- ❌ 工具结果缓存（同一参数不重复调用）
- ❌ 多 Agent 协作或对话
- ❌ 对话历史持久化（仍和第一阶段一样，退出即丢失）

## 验收标准

- AC1: 用户输入"在 src/main.ts 中找到 main 函数并告诉我它做了什么"，Agent 自动调用 grep 定位函数，再调用 read_file 查看内容，最后给出分析。整个过程无需用户手动干预
- AC2: 用户输入"在 src 目录下创建一个 hello.txt，内容是 Hello World"，Agent 自动调用 write_file 创建文件。用户可通过 ls 验证文件存在
- AC3: 用户输入一个简单问题"你好"，Agent 不触发任何工具，直接文字回复
- AC4: Agent 连续 5 次调用工具后仍未完成任务，自动停止并给出总结
- AC5: 用户输入 `/plan` 后，Agent 切换为规划模式。用户说"帮我看看这个项目结构"，Agent 只用只读工具调研并给出计划
- AC6: 用户输入 `/do` 后，Agent 切换回执行模式，可以使用所有工具
- AC7: Agent 执行过程中按 Ctrl+C，循环终止但不退出程序
- AC8: 模型一次返回 2 个只读工具调用时，两个工具并发执行而非串行等待
- AC9: 模型连续 2 次调用不存在的工具，Agent 终止循环并报告错误
- AC10: Agent 执行完成后，能继续下一轮对话（上下文保持）
- AC11: 所有现有测试继续通过，`npx tsc --noEmit` 零类型错误
